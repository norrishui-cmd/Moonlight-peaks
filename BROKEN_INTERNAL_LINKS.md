# Broken Internal Links Audit

## Method

Extracted every `href="..."` string literal (not dynamic template-literal hrefs, which are
covered separately below) across all `.astro` and `.ts` files under `src/`, resolved each against
the actual production build output (`dist/`) generated by this session's `npm run build`, and
checked each resolves to a real file (`index.html`, a direct `.html`, or a direct file for JSON/XML
endpoints).

## Result

**82 unique static internal links checked. 0 broken.**

Checks performed and all clear:

- **Links to removed routes:** 0. Cross-checked the 46 routes removed in `SEO_ROUTE_DIFF.md`
  against all 172 source files in `src/` â€” zero references found anywhere (not in `related`
  arrays, breadcrumbs, hub pages, structured data, or static data files). These routes were never
  linked to begin with; they were sitemap/seoGraph-only leaf pages.
- **Links to nonexistent routes:** 0, after fixing the 3 genuine cases found this pass (see
  below).
- **Malformed relative links:** 0 (checked for stray spaces, `undefined`, `null`, `NaN` embedded
  in href strings).
- **Duplicate slashes:** 0.
- **Incorrect locale routes:** 3 found and fixed (see below) â€” not literally "duplicate slash" or
  "malformed" issues, but locale-routing bugs surfaced by the sitemap reconciliation in Phase 7,
  which is the same family of problem this phase is meant to catch.
- **Links to old programmatic URLs:** 0 â€” the only programmatic URLs that changed
  (`/characters/{id}/gifts`) were never referenced by any static link in the first place (see
  `SEO_ROUTE_DIFF.md`).

## Issues found and fixed this pass (not "static href" bugs, but same risk category)

These weren't caught by the static-href scan above (they're driven by data arrays, not literal
`href="..."` strings), but were caught by the Phase 7 sitemap-to-build reconciliation and are
exactly the class of bug this phase exists to catch:

1. **`src/i18n/characters-data.ts`** had a stale `logan` key (in all three of `es`/`ja`/`zh`) left
   over from the Loganâ†’Brook character correction made in an earlier session. The sitemap had
   already been updated to expect `/es|ja|zh/characters/brook`, but the translation data object
   still kept the old key and old (pre-correction) content describing "Logan" as a romanceable
   werewolf. **Fixed:** renamed the key to `brook` in all three locales and rewrote the content to
   match the corrected facts (mayor, not romanceable, Howling Hammer hours).
2. **`src/pages/es/characters/index.astro`, `ja/characters/index.astro`, `zh/characters/index.astro`**
   each had a second, independent hardcoded array (`translatedSlugs`) also still listing `'logan'`.
   This caused a hard build failure (`undefined` translation lookup) once the key was renamed in
   step 1, until fixed. **Fixed:** updated all three arrays to `'brook'`.
3. **`src/pages/sitemap.xml.ts`** listed `/es/environment`, `/ja/environment`, `/zh/environment` in
   `staticPaths`, but no such page files exist in any locale. **Fixed:** removed the three
   nonexistent paths from the sitemap rather than creating new pages (out of scope for this
   validation-only pass).
4. **`src/pages/tools/romance-match-quiz.astro`** included `'logan'` as a quiz result option,
   presenting him as a distinct, romanceable match â€” stale from before the Loganâ†’Brook correction
   (Logan is now understood as a family name, not a person, and Brook â€” the actual mayor behind
   the old assumption â€” is explicitly not romanceable). **Fixed:** removed the `logan` entry from
   the quiz's `cast` and `results`, and re-pointed the two quiz options that scored toward it to
   the already-valid `orlock` result instead.

All four were caused by the same root event (an earlier session's Loganâ†’Brook correction that
wasn't fully propagated) â€” not by this validation pass's semantic-gate work, and not something
this task was originally scoped to find. They were caught incidentally by the Phase 7 sitemap
reconciliation and fixed because leaving them would mean submitting URLs to search engines that
404, and leaving the quiz bug would mean live incorrect content on the site.

## Dynamic (data-driven) href patterns â€” separately verified

Template-literal hrefs like `href={`/characters/${c.id}`}` aren't literal strings the static scan
above can check, but they're inherently safe here: every such pattern iterates over the same
`characters` (or `locations`, `families`, etc.) array used to generate the actual pages via
`getStaticPaths`, so the link target and the page source are the same array by construction â€” they
cannot drift apart. Confirmed by the production build succeeding for all 410 HTML pages with zero
runtime errors.
